# Troubleshooting

---

## bootstrap.sh fails at ansible version check

**Symptom**: `scripts/bootstrap.sh` exits with an error around line 15.

**Cause**: Line 15 contains an em-dash (`–`) instead of `--` in a pacman flag — a pre-existing character encoding bug.

**Fix**: If Ansible was still installed despite the error, verify with:
```bash
ansible --version
```
If it's present, skip the script and run the playbook directly. If not, install manually:
```bash
sudo pacman -S ansible
```

## awww-daemon crashes on startup

**Symptom**: `awww query` returns an error; wallpaper is not set after login.

**Cause**: `awww-daemon` segfaults if started before the Wayland session is fully initialized.

**Mitigation already in place**: `exec-once = bash -c 'sleep 3 && awww-daemon'` delays startup by 3 seconds. `theme-switch.sh` also auto-starts the daemon if it's not running when the script fires.

**Fix**: If it still fails, run `theme-switch.sh` manually after the desktop is stable:
```bash
~/.local/bin/theme-switch.sh
```

## YouTube authentication (two separate steps required)

**Symptom**: `:r` in youtube-tui either fails, plays without your recommendations, or shows no subscriptions feed.

**Cause**: There are two independent auth mechanisms:

1. **yt-dlp / yt-recommended** (mpv recommended feed) — uses Zen Browser cookies
2. **youtube-tui itself** — uses a Google device code

**Fix for yt-dlp**: Log into YouTube in Zen Browser, then regenerate the config:
```bash
ansible-playbook ansible/configurations/arch.yml --tags youtube_tui -K
```

**Fix for youtube-tui**: Inside the app, open Settings → Login and follow the device code at `https://google.com/device`.

## yt-dlp config not updated after Zen login

**Symptom**: `yt-recommended` fails to authenticate even after logging into Zen Browser.

**Cause**: `~/.config/yt-dlp/config` is generated by Ansible at playbook time — it is not stowed and does not update automatically.

**Fix**: Re-run the youtube_tui role:
```bash
ansible-playbook ansible/configurations/arch.yml --tags youtube_tui -K
```

## Rofi click-to-exit not working

**Symptom**: Clicking outside the rofi launcher visible box doesn't close it.

**Note**: `click-to-exit: true` is silently ignored on Wayland (rofi-wayland). The working implementation uses:
1. `kb-cancel: "Escape,MousePrimary"` in `config.rasi` — binds left click to cancel
2. A fullscreen 100%×100% transparent `window` in `theme.rasi` — ensures rofi's layer surface captures all pointer events across the full screen
3. Centered visible box via spacer elements

If this breaks after a rofi update, check that `theme.rasi` still has `window { width: 100%; height: 100%; }` and that `config.rasi` still has the `kb-cancel` binding.

Do **not** use `-normal-window` flag — it breaks this behavior.

## Zen Browser windowrules don't trigger

**Symptom**: A windowrule for class `zen` doesn't fire when opening a URL in Zen.

**Cause**: If Zen Browser is already running, `zen-browser URL` opens a tab in the existing window — no new window event is emitted, so no windowrule triggers.

**Fix**: Kill the existing Zen process first, then open the URL:
```bash
pkill -x zen-browser && zen-browser URL
```

## Ansible `import_tasks` silently skips tagged tasks

**Symptom**: Running `ansible-playbook --tags <tag>` does nothing even though the tasks have the correct tag.

**Cause**: Roles included via `include_role` (dynamic) must use `include_tasks` internally — not `import_tasks`. Using `import_tasks` with explicit tags inside a dynamically included role causes those tasks to silently not execute when running without `--tags`.

**Fix**: In any role's `tasks/main.yml`, use:
```yaml
- include_tasks: setup.yml
```
Not:
```yaml
- import_tasks: setup.yml  # broken with include_role + tags
```

## Hardcoded `admin` username causes failures on other systems

**Symptom**: Playbook fails during `core_packages` role; or Neovim terminal splits don't open on project load.

**Cause**: Two files hardcode `/home/admin/`:
- `ansible/configurations/roles/core_packages/tasks/install.yml` — yay build path
- `dotfiles/home/.config/nvim/lua/config/autocmds.lua` — project directory detection
- `dotfiles/home/.config/hypr/hyprland.conf` — workspace autostart commands

**Fix**: See [Customization → Username](customization.md#username).

## Theme colors not updating after wallpaper change

**Symptom**: Waybar or Neovim show old colors after `theme-switch.sh` runs.

**Possible causes and fixes**:

1. **Waybar**: Waybar reloads on `SIGUSR2`. Check if the process is running: `pgrep waybar`
2. **Neovim**: Hot-reload uses the Neovim socket. If Neovim was started before the socket was set up, it may miss the update — restart Neovim or run `:colorscheme mini-base16` manually.
3. **matugen failure**: Run `matugen image <wallpaper-path>` manually to see if it errors.
